# Camera (Issue #49)

## Summary

Added basic camera logic to the `core` module: a `ProjectionType` enum, a `CoordinateSystem` enum, six view/projection matrix factories on `Mat4f`, and a `Camera` class that maintains view, projection, view-projection, and inverse-view matrices plus the camera's x/y axes in world space, all rebuilt explicitly via `UpdateMatrices()`.

## Design Decisions

### UpdateMatrices() as the sole rebuild point
Setters only store the value; no matrix recomputation happens until `UpdateMatrices()` is called explicitly. This matches the issue requirement and avoids rebuilding matrices multiple times when several properties are changed in sequence.

### screen_center_ as (half_width, half_height)
The issue lists "screen center" as a property. Interpreted as `Vec2f(half_width, half_height)` of the viewport, which gives:
- `aspect = screen_center.x / screen_center.y` (perspective)
- Full ortho extents: `width = 2 * screen_center.x`, `height = 2 * screen_center.y`

Guard: if `screen_center.y == 0`, aspect defaults to 1 to avoid division by zero.

### LookAt: RH vs LH differ only in z_axis sign
- RH: `z = normalize(eye - target)` (camera looks down −Z, OpenGL convention)
- LH: `z = normalize(target - eye)` (camera looks down +Z, DirectX convention)

Both variants share the same x/y construction and the same matrix form.

### Projection NDC depth ranges
- RH: z in [−1, 1] (OpenGL)
- LH: z in [0, 1] (DirectX)

### view_proj_ = proj_ * view_
`Vec3f * Mat4f` is implemented as row·M = M*v (column-vector convention, verified from source). So:
- `v * view_proj_` = `view_proj_ * v` = `proj_ * view_ * v`
- This gives the correct `clip = proj * view * vertex` transform.

### x_axis / y_axis from view matrix rows
After LookAt, rows 0 and 1 of the view matrix are the camera's right and up axes expressed in world space. Extracted directly from the matrix rather than recomputing from position/target.

## For Future Features

- `Camera` is currently not connected to the rendering pipeline — it produces matrices but nothing binds them to a constant buffer. Once a constant buffer layout for camera data is defined, `Camera::UpdateMatrices()` can be followed by `cb->Fill(&view_proj_)`.
- `fov_` is always treated as `fov_y`. A separate `fov_x` or an `aspect` override might be useful when the viewport is non-square and fov_x is the authoring intent.
- `screen_center_` as half-extents is an unusual convention. If asymmetric frustum shifts (lens shift / jitter) are needed later, a separate `Vec2f lens_shift_` property (in NDC) can be added to the projection matrices without changing the existing interface.
