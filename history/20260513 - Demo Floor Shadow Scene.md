# Demo Floor Plane and Shadow Scene Setup

**Date:** 2026-05-13
**Branch:** feat/demo-floor-shadow-scene
**Issue:** #152

## Summary

Added a procedural floor plane to the demo, a `MeshTemplate` constructor for pre-built meshes, a `CreatePlaneMesh` geometry utility, and tuned demo lighting so all shadow types (CSM, spot, cube) are visually demonstrated.

---

## Changes

### `src/renderer/GeometryUtils.h/.cpp`

Added `CreatePlaneMesh(video, half_size, material)`:
- 4 vertices in the XZ plane at Y=0 with upward normals `{0,1,0}`.
- UVs tiled at `1 tile per 5 world units` (tiles = half_size / 5).
- 2 triangles: `{0,1,2, 0,2,3}` (CCW winding viewed from above).
- Takes ownership of `material`; returns a heap-allocated `RenderableMesh*`.
- Forward-declared `Material` and `RenderableMesh` in the header to avoid heavy includes.

### `src/game/MeshTemplate.h/.cpp`

Added `explicit MeshTemplate(renderer::RenderableMesh* mesh)`:
- Takes ownership of a pre-built mesh, bypasses file loading.
- Registers in the `Resource` cache with a synthetic key (`"__proc_0"`, `"__proc_1"`, …) generated by a local static counter, ensuring each instance gets a unique key and is correctly removed on `Release()`.
- Sets `initialized_ = true` immediately (no load can fail).

### `src/app/main.cpp`

**Floor plane:**
- Grey concrete material via `MaterialDesc().SetDiffuseColor({0.45f, 0.45f, 0.45f})`.
- `CreatePlaneMesh(video, 30.f, plane_mat)` — 60×60-unit plane.
- Wrapped in `MeshTemplate` + `GameMesh`, placed at world identity (Y=0).

**Demo mesh positions:** lifted from `Y=0` to `Y=3` so meshes float above the floor and their shadows fall clearly onto it.

**Global light direction:** changed from `(-1,-1,-0.5).Normalized()` to `(-0.4,-0.8,-0.3).Normalized()` — shallower oblique angle produces longer, more visible shadows.

**New CircleSpotLight:**
- Position: `{0, 10, 5}`, direction: `{0,-1,-0.3}.Normalized()`.
- `outer_angle=0.4`, `range=25`.
- Demonstrates spot-light shadow rendering on the floor.

## Decisions and Rationale

### `MeshTemplate(RenderableMesh*)` constructor uses a synthetic registry key

`core::Resource` always registers itself in the static map. A static counter provides unique keys (`__proc_N`) for procedural meshes that don't have a file path. This ensures correct lifecycle management (`Release()` can erase the key from the map) without touching the base class.

### `CreatePlaneMesh` takes ownership of `Material*`

Consistent with `RenderableMesh::AddSubmesh(unique_ptr<GeometryData>, unique_ptr<Material>)`. The caller creates the material and passes it — no double-ownership issues if done in a single expression chain.

### Floor at Y=0, meshes at Y=3

A 3-unit lift gives enough vertical separation for shadows to land visibly on the plane at typical camera angles, while keeping the scene compact.

---

## Key Points for Future Features

- `CreatePlaneMesh` does not compute tangents or binormals (zeroed). Normal-mapped materials on the floor will look wrong until tangent computation is added.
- The `__proc_N` key convention is an internal detail; avoid registering procedural MeshTemplates under meaningful path names to prevent accidental cache hits.
- Spot light shadows require the spot light's `cast_shadow` to be true (default). The pool must have a free slot — verify the config allows at least one 2D shadow map.
