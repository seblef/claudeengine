# Demo Scene (Issue #87)

## Summary

Added a full end-to-end demo scene that validates the deferred lighting
pipeline with two mesh types, two materials, and one instance of each
light type.

## Changes

### `src/app/main.cpp`

All changes are confined to this one file.

**`BuildSphere()` helper (anonymous namespace)**

Generates a UV sphere `GeometryData` with 32 stacks × 64 rings. Each vertex
carries a full `Vertex3D` record:

- `position` = (sin_phi × cos_theta, cos_phi, sin_phi × sin_theta) — Y-up sphere
- `normal` = position (outward for a unit sphere)
- `tangent` = d(pos)/d(theta) normalized = (−sin_theta, 0, cos_theta) — direction
  of increasing U; degeneracy at poles handled by returning (1,0,0)
- `binormal` = d(pos)/d(phi) = (cos_phi × cos_theta, −sin_phi, cos_phi × sin_theta)
  — direction of increasing V
- `uv` = (j/rings, i/stacks)

Triangle winding: two CCW triangles per quad when viewed from outside, generated
as `(k0, k0+1, k1)` and `(k0+1, k1+1, k1)` — verified geometrically.

Vertex count: 33 × 65 = 2145 (within uint16_t range).
Triangle count: 32 × 64 × 2 = 4096.

**Materials**

- Cube: `MaterialDesc().SetDiffuse("demo.png").SetShininess(64)`
- Sphere: same diffuse but `SetAmbientColor({0.05,0.05,0.1})`, which makes
  `IsEmissive()` return true and routes the sphere through the emissive pass
  each frame, producing a subtle self-lit blue ambient contribution.

**Lights**

| Light | Parameters |
|---|---|
| `GlobalLight` | direction={-1,-1,-0.5} normalized (incoming ray direction), warm white color, ambient=(0.05,0.05,0.08) |
| `OmniLight` | position={-2,1,0}, radius=3, orange, intensity=2 |
| `CircleSpotLight` | position={0,4,0}, direction=down, inner=20°, outer=30°, range=8, cool blue |
| `RectangleSpotLight` | position={3,2,0}, direction={-1,-1,0} normalized, h=25°, v=15°, range=8, neutral white |

All lights are local variables in `main()`; `Enqueue()` adds `this` to
`LightRenderer`'s per-frame list, which is cleared by `EndRender()` at the
end of each `Renderer::Update()`. Safe with local variable lifetime.

**Debug mode keys**

- `0` or `Escape`: restore full 4-phase pipeline
- `1`: albedo G-buffer channel
- `2`: normal G-buffer channel
- `3`: specular G-buffer channel
- `4`: depth (linearized)

**Camera and scene scale**

- Camera moved from {0,0,50} to {0,0,12} to match the smaller scene
- `kHalf` reduced from 5 → 1.5 (cube ±1.5 in each axis, 3 units across)
- Cube placed at {−3,0,0} (rotating around its own Y axis)
- Sphere (unit radius) placed at {3,0,0} (static)

## Decisions and Rationale

**BuildSphere in main.cpp anonymous namespace**: The existing `CreateSphere`
in `GeometryUtils` fills only position (by design — it's used for light
volume meshes). Adding a full-TBN variant to `GeometryUtils` would add surface
rendering concerns to a file designed for volume primitives. Keeping it in
`main.cpp` keeps the scope isolated to the demo, as the issue allows.

**Sphere winding verification**: The CCW convention was derived analytically.
At the equator (phi=π/2, theta=0), looking from outside in the −X direction:
tangent T = (0,0,1) = "right", binormal B = (0,−1,0) = "down". Triangle
(k0, k0+1, k1) in 2D screen space is (0,0)→(1,0)→(0,1) which is CCW.

**Sphere ambient for emissive testing**: The issue requires "emissive texture:
non-zero on one of the objects." Using `ambient_color` instead of a custom
emissive texture avoids creating a new texture asset and is sufficient to
exercise the emissive pass (IsEmissive() checks `ambient_color.rgb != 0`).

## Output to Keep in Mind

- `BuildSphere` uses `stacks=32, rings=64` producing 2145 vertices — fits
  comfortably in uint16_t but is close to the limit for very dense spheres.
  Future large spheres may need uint32_t indices.
- The GlobalLight `direction` convention: the vector points in the incoming
  light direction (sun→surface); the shader does `L = normalize(-direction)`.
  The header comment says "toward the light source" which is the opposite
  convention — a known discrepancy worth clarifying in a follow-up.
