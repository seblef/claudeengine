## Decisions and rationale

### `TransformTool` absorbs both gizmo and picking

When a transform tool is active it replaces `SelectionTool` as `active_tool_base_`. To preserve the ability to select a different object by clicking outside the gizmo, `TransformTool::OnRender` also calls `PickObjectAt` and `DrawSelectedBBox`, guarded by the same `ImGuizmo::IsOver()`/`IsViewManipulateHovered()` checks as `SelectionTool`.

### Spurious-pick guard uses previous-frame state

`gizmo_was_using_` is snapshotted into `was_using` before the gizmo block updates it. This mirrors the old approach where the viewport passed the previous-frame value via `EditorToolContext::gizmo_was_using`.

### `SetActiveTool(nullptr)` resets to SelectionTool

Passing `nullptr` to `EditorViewport::SetActiveTool(EditorToolBase*)` now silently resets `active_tool_base_` to `&selection_tool_`. This lets `EditorWindow` cleanly switch back to selection without needing a raw pointer to the viewport's internal `SelectionTool`.

### `EditorTool` enum values `kTranslate/kRotate/kScale` kept in viewport

The old `active_tool_` field and `SetActiveTool(EditorTool)` inline accessor are left untouched; the viewport refactor that removes them is tracked in a separate issue (as noted in the acceptance criteria).

### ImGui/ImGuizmo include order

`<imgui.h>` must precede `<ImGuizmo.h>` in every translation unit since ImGuizmo.h uses ImVec2, ImDrawList, etc. declared by imgui.

## Keep in mind for next features

- `EditorTool::kTranslate/kRotate/kScale` are no longer needed by the viewport gizmo code — a dedicated viewport-refactor issue should remove them and the old `SetActiveTool(EditorTool)` path.
- `EditorToolContext::gizmo_was_using` is now always `false` when `SelectionTool` runs (no gizmo can fire while SelectionTool is active). The field can be cleaned up once the two-track tool system is fully unified.
