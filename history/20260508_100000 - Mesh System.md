# Mesh System

**Date:** 2026-05-08
**Branch:** feat/mesh-system
**Issue:** #70

## Summary

Introduced the first concrete renderable type: `Mesh`, `MeshInstance`, and `MeshRenderer`. A `Mesh` bundles a `GeometryData` and a `Material`. A `MeshInstance` is a `Renderable` that references a `Mesh` and enqueues itself into `MeshRenderer` via `Enqueue()`. `MeshRenderer` is a singleton batch renderer that sorts instances by material then by mesh before issuing draw calls. The global `Renderer` creates and destroys `MeshRenderer` and drives it each frame. The cube demo was rewritten to use the new system.

## Changes

### New: `src/renderer/Mesh.h` / `Mesh.cpp`

Non-owning container pairing a `GeometryData*` and a `Material*`. Provides getters and setters. The caller is responsible for keeping both alive for the lifetime of the `Mesh`.

### New: `src/renderer/MeshInstance.h` / `MeshInstance.cpp`

Concrete `Renderable` subclass. Constructor takes a `Mesh*`, an initial world matrix, and an `always_visible` flag. Passes `model->GetGeometryData()->GetBBox()` as the local bounding box to the `Renderable` base class. `Enqueue()` forwards `this` to `MeshRenderer::Instance().AddInstance()`.

### New: `src/renderer/MeshRenderer.h` / `MeshRenderer.cpp`

Singleton batch renderer. Inherits from `core::Singleton<MeshRenderer>` (first, offset 0) and `ObjectRenderer<MeshInstance>` (second). `Singleton<MeshRenderer>` is first to ensure the `static_cast<MeshRenderer*>(this)` in the base constructor is a zero-offset operation.

`Render()` iterates the sorted instance list, skips `Material::Set()` when the material pointer is unchanged, skips `GeometryData::Set()` when the mesh pointer is unchanged. Material change is the more expensive state switch, so it is the primary sort key.

### New: `data/shaders/glsl/geometry/mesh_base_vs.glsl` / `mesh_base_ps.glsl`

Vertex shader: transforms `in_position` through `world` (UBO binding 1) then `view_proj` (UBO binding 2) and passes `in_uv` through.
Fragment shader: samples `u_diffuse` at binding 0.

### Modified: `src/renderer/Renderer.h`

Added public `~Renderer()` declaration.

### Modified: `src/renderer/Renderer.cpp`

- `#include "renderer/MeshRenderer.h"` added.
- Constructor ends with `new MeshRenderer(video_)`.
- `~Renderer()` calls `MeshRenderer::Shutdown()`.
- `Update()` calls `PrepareRender()`, `Render()`, `EndRender()` on `MeshRenderer::Instance()` after filling the scene CB.

### Modified: `src/renderer/CMakeLists.txt`

Added `Mesh.cpp`, `MeshInstance.cpp`, `MeshRenderer.cpp` to the `renderer` static library.

### Modified: `src/app/main.cpp`

- Removed `abstract/Shader.h` include and manual shader creation/activation/release.
- Added `renderer/Mesh.h` and `renderer/MeshInstance.h` includes.

## Decisions & Rationale

### Mesh does not own GeometryData or Material
Non-owning pointers, consistent with how `VideoDevice*` is handled throughout the codebase. Lifetime is the caller's responsibility.

### MeshInstance passes GetBBox() from GeometryData
`GeometryData` already computes the AABB from vertex positions at construction. Passing it directly to `Renderable` gives correct world-space culling data immediately.

### MeshRenderer created/destroyed by Renderer
The issue spec states "The singleton should be instantiated in the global renderer constructor." A matching destructor in `Renderer` keeps the lifetime symmetric. `main.cpp` needs no change to manage `MeshRenderer`.

### Multiple inheritance order: Singleton first
`Singleton<MeshRenderer>` is listed before `ObjectRenderer<MeshInstance>`. The `Singleton<T>` base subobject is then at offset 0, making `static_cast<MeshRenderer*>(this)` in the Singleton constructor correct without any offset adjustment.

### Sort by material then mesh
`PrepareRender()` (from `ObjectRenderer<T>`) sorts on `(material*, mesh*)`. `Render()` skips `Material::Set()` when consecutive instances share the same material, and skips `GeometryData::Set()` when they also share the same mesh. Material change is more costly than geometry rebind (texture uploads vs. buffer binding), so it is the primary sort key.
