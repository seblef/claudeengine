# VehiclePanel — resource editor for .vehicle.yaml (issue #717)

## What was done

Implemented `editor::VehiclePanel`, the first concrete `IResourcePanel` subclass,
registered in `EditorWindow` for the `.vehicle.yaml` extension.

## Key decisions

### YAML format

The panel uses the nested format specified in the issue:

```yaml
body_mesh: …
front_wheel_mesh: …
rear_wheel_mesh:  …
physics:
  mass: …
  suspension:
    front: { min_length, max_length, stiffness, damping }
    rear:  { … }
wheels:
  front_left:  { position: [x, y, z] }
  …
```

`VehicleTemplate.cpp` was updated with backward-compatible parsing: if a `physics:`
node exists the new format is assumed; otherwise the old flat format is used. No
existing `.vehicle.yaml` files existed in the repository, so the migration is clean.

### Per-axle suspension

The panel edits suspension per axle (front / rear), not per wheel. Both axle wheels
always share the same values. This matches the issue spec and avoids exposing 8
redundant sliders.

### is_driven / is_steered

The VehiclePanel YAML does not expose `is_driven` / `is_steered`. When parsing the
new format, `VehicleTemplate.cpp` hard-codes the conventional layout:
front wheels steered, rear wheels driven. This can be extended later.

### Combined preview

A 320×240 offscreen preview renders body + 4 wheel meshes. Wheel `MeshInstance`
world matrices are updated each frame from the current `vehicle_desc_` positions,
so the preview stays in sync while the user edits wheel positions.

To support multi-mesh rendering the `PreviewRenderer::Render()` was overloaded with
a `std::span<MeshInstance* const>` variant. The implementation enqueues all instances
before calling `MeshRenderer::PrepareRender()` — consistent with how MeshRenderer's
queue model works.

### ImGuizmo gizmo

When a wheel row is focused (via `DragFloat3` or focus click), an `ImGuizmo::Manipulate`
translate gizmo is drawn on top of the combined preview. Matrix conventions follow
the same pattern used in `EditorViewport` (transpose for ImGuizmo's row-vector
convention, transpose back on readout).

X-mirror is enforced both in the gizmo handler and in the `DragFloat3` row handler:
editing FL.x sets FR.x = -FL.x; editing RL.x sets RR.x = -RL.x.

### Place in Scene

The "Place in Scene" button fires a callback registered by `EditorWindow` in the
factory lambda. The callback loads the `VehicleTemplate`, creates a `GameVehicle`,
positions it at the editor camera's focus point, and pushes a `PlaceObjectCommand`
for undo support. `VehiclePanel` itself never imports `EditorScene` or
`EditorCommandHistory` — dependency inversion via `std::function`.

## Output to keep in mind

- `VehiclePanel` follows the `IResourcePanel` lifecycle: `Draw()` each frame,
  `IsDirty()` drives the tab dirty indicator, `Save()` flushes to disk.
- The `ResourceBrowser` handles Ctrl+S by calling `panel->Save()` on the active tab —
  VehiclePanel does not register its own key binding.
- The `on_place_in_scene_` callback is set AFTER construction in the factory lambda,
  so the button is disabled (greyed out) until the lambda runs and provides the callback.
