# Material Base

**Date:** 2026-05-06
**Branch:** feat/material-base
**Issue:** #67

## Summary

Added basic material support backed by per-slot textures. A material holds one texture per normalized slot (diffuse, normal, specular, emissive, environment). Unspecified slots fall back to 1×1 default textures. The demo now constructs a material from a YAML file rather than binding a raw texture directly.

## Changes

### New: `src/renderer/TextureSlot.h`

Defines `renderer::TextureSlot` (scoped enum, int underlying type) and `kTextureSlotCount = 5`. Slot indices map directly to OpenGL texture unit indices used in `Set()`.

### New: `src/renderer/Material.h` and `Material.cpp`

**Constructors:**
- `Material(VideoDevice*)` — loads all five defaults.
- `Material(TextureArray, VideoDevice*)` — AddRefs provided textures, fills null slots with defaults.
- `Material(const std::string& name, VideoDevice*)` — loads `data/materials/<name>` via `core::LoadYamlFile`, falls back to all defaults on error.
- `Material(const YAML::Node&, VideoDevice*)` — loads from a pre-parsed YAML node.

**Methods:**
- `Set()` — binds each texture to `GL_TEXTURE0 + slot` via `Bind(i)`.
- `GetTexture(TextureSlot)` — per-slot accessor.
- `GetTextures()` — returns `const TextureArray&` (the full array).
- `SetTexture(TextureSlot, Texture*)` — replaces a slot, releases old ref, AddRefs new.

**Reference counting:** Material owns one reference per texture (acquired via `CreateTexture` or `AddRef`). Destructor calls `Release()` on every non-null slot.

**YAML format:**
```yaml
textures:
  diffuse: <name relative to data/textures/>
  normal: ...
  specular: ...
  emissive: ...
  environment: ...
```
All fields optional.

### New: default textures (`data/textures/default/`)

Five 1×1 pixel PNG files generated with stdlib Python (no external deps):
| File | RGBA |
|------|------|
| `diffuse.png` | (255, 255, 255, 255) — white |
| `normal.png` | (128, 128, 255, 255) — up in tangent space |
| `specular.png` | (0, 0, 0, 255) — black |
| `emissive.png` | (0, 0, 0, 255) — black |
| `environment.png` | (0, 0, 0, 255) — black |

### New: `data/materials/demo.yaml`

```yaml
textures:
  diffuse: demo.png
```

### Modified: `src/renderer/CMakeLists.txt`

Added `Material.cpp` to the renderer static library. No new CMake dependencies — `yaml-cpp` is already available transitively via `abstract → core → yaml-cpp`.

### Modified: `src/app/main.cpp`

Replaced `abstract::Texture* tex = video->CreateTexture("demo.png")` with `renderer::Material material("demo.yaml", video)`. In the render loop, `material.Set()` replaces `tex->Bind(0)`. The explicit `tex->Release()` cleanup is gone — Material's destructor handles lifetime.

## Decisions & Rationale

### `yaml-cpp` is not added explicitly to renderer's CMakeLists

It propagates through `abstract (INTERFACE) → core → yaml-cpp`. Adding it explicitly would create a redundant dependency.

### TextureArray constructor AddRefs

When the caller already holds a texture reference, the material must AddRef to take co-ownership. The caller is still responsible for their own Release.

### LoadFromYaml avoids double-AddRef

`CreateTexture` returns with ref_count = 1. In `LoadFromYaml`, after releasing the default, the returned pointer is directly stored (no additional AddRef). This keeps the semantics clean: "we own exactly one ref per slot."

### Default textures as pre-generated PNG files

Generated at asset-creation time rather than at runtime. This avoids a runtime dependency on any image-writing library and keeps startup fast.

## Notes for Future Contributors

- To add a new texture slot, update `TextureSlot.h` (increment `kTextureSlotCount`, add enum value), `kDefaultPaths[]`, `kSlotMap[]` in `Material.cpp`, and create the corresponding default texture file.
- The YAML key names in `kSlotMap` must match the YAML material file keys exactly.
- `Material` is non-copyable. If move semantics are needed later, implement a move constructor that nulls the source's texture pointers.
