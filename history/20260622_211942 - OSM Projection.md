# OSM Importer: WGS84 → Local XZ Projection (#746)

## What was done

Implemented `tools/osm_importer/projection.py` — replaced the placeholder `project()` stub with a `Projector` class providing equirectangular projection from WGS84 lat/lon to a flat local XZ plane in metres.

## API implemented

```python
class Projector:
    def __init__(self, bbox: tuple[float, float, float, float])
    def project(self, lat: float, lon: float) -> tuple[float, float]
    def map_size(self) -> float
```

- `bbox` = `(lat_min, lon_min, lat_max, lon_max)` — matches the fetcher convention
- `project()` returns `(x, z)` in metres, origin at bbox centre
- `map_size()` returns the longest bbox axis in metres, rounded up to the nearest 10

## Key decisions

| Decision | Rationale |
|---|---|
| Class over bare function | Precompute `cos(lat_mid)` once; bbox midpoint stored for reuse by roads and buildings |
| `METRES_PER_DEGREE = 111320.0` | Standard WGS84 approximation per issue spec |
| `math.ceil(… / 10) * 10` for rounding | Guarantees map size is always ≥ actual span, preventing clipping |
| No external deps | Issue spec: pure `math` stdlib only |

## Verification

Sanity-checked against a Paris bbox:
- Centre `(48.86, 2.345)` → `(0.00, 0.00)` ✓
- 1 m north/east offset → `(1.00, 1.00)` ✓
- SW/NE corners symmetric: `±1099 x`, `±1113 z` ✓
- `map_size()` = `2230.0` m (longest axis = 2226.4 m → rounded up) ✓

## Output to keep in mind

- `Projector.__init__` expects bbox in `(lat_min, lon_min, lat_max, lon_max)` order — roads.py and buildings.py must follow this convention
- `map_size()` result is the canonical map dimension for the `.map.yaml` writer (#749)
- The old standalone `project()` function was removed; any future callers must use the class
