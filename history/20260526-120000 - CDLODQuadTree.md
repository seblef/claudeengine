# CDLODQuadTree: LOD selection with configurable triangle budget

**Issue:** #284

## Summary

Implemented the `TerrainPatch` data struct and the `CDLODQuadTree` class that
drives terrain LOD selection.  Given a `Camera` and a triangle budget,
`Select()` frustum-culls the quadtree and fills a flat list of
`TerrainPatch` descriptors ready for GPU submission.

## Design decisions

### Flat node vector

All quadtree nodes live in `std::vector<Node> nodes_`; children are
addressed by integer index rather than pointer.  This eliminates heap
fragmentation from thousands of tiny allocations and makes cache-line
traversal sequential during Select().  The index-based approach also
sidesteps the iterator-invalidation hazard that would occur if child nodes
were inserted while a parent reference was live.

### LOD distance derivation from `max_triangles`

The sole public knob for triangle budget is `max_triangles` passed to
`Select()`.  LOD switch distances are computed each call (cheap ‚Äî one
`sqrt` plus lod_count multiplications) so the budget can change frame to
frame without rebuilding the tree.

**Formula derivation:**  
For a camera at the origin, LOD l contributes triangles from a ring between
radii `d_{l-1}` and `d_l = d_0 * 2^l`.  With patch world size
`W_l = patch_size * 2^l * mpt` and `T = 2 * patch_size¬≤` triangles per
patch, the triangle count per ring is:

```
ring_area   = œÄ * (d_l¬≤ - d_{l-1}¬≤) = 3œÄ * d_0¬≤ * 4^(l-1)
patch_area  = W_l¬≤ = patch_size¬≤ * 4^l * mpt¬≤
patches     = ring_area / patch_area  = 3œÄ * d_0¬≤ / (4 * patch_size¬≤ * mpt¬≤)
tris_ring_l = patches * T            = 3œÄ * d_0¬≤ / (2 * mpt¬≤)   [independent of l]
```

Summing `lod_count` rings (treating LOD 0 as a disk) gives approximately:

```
total ‚âà lod_count * 2œÄ * (d_0 / mpt)¬≤  =  max_triangles
‚Üí d_0 = mpt * sqrt(max_triangles / (2œÄ * lod_count))
‚Üí d_l = d_0 * 2^l
```

The approximation is intentionally conservative (the 2œÄ factor slightly
under-estimates vs. the exact 3œÄ/2 ring formula), giving a small safety
margin so the actual count stays at or below budget.

### AABB height bounds

For **leaf nodes** (LOD 0): all `(patch_size+1)¬≤` vertex heights are sampled
via `TerrainData::GetHeight` during `Build()`.  This is exact ‚Äî the bounds
match exactly what the vertex shader will use.

For **interior nodes**: the Y bounds are the union of their four children's Y
bounds, propagated bottom-up.  This is O(n) total work and produces
conservative AABBs suitable for frustum culling.  XZ bounds come directly
from the grid position and require no height sampling at all.

`TerrainData` was not modified; only its public API (`GetHeight`,
`GetTexelWidth/Height`, `GetWorldWidth`) is used.

### Morph range

Each patch's `morph` field ramps from 0 to 1 over the last 20 % of its LOD
distance range `[0, d_l]`, i.e., in the interval `[0.8 * d_l, d_l]`.  The
vertex shader is expected to lerp vertex positions toward their LOD l+1
counterpart using this factor, eliminating the visible pop at LOD boundaries.
The coarsest LOD (lod_count-1) always emits `morph = 0` because there is no
coarser level to blend toward.

### Terrain-edge partial tiles

The terrain width/height may not be an exact multiple of the top-level tile
size.  Edge children whose `(cgx, cgz)` falls outside the computed child-LOD
grid simply get no slot (`children[i] = -1`).  `SelectNode` skips
`children[i] < 0` entries, so partial boundary tiles are handled transparently
without any special-casing in `Select()`.

## Output to keep in mind

- `CDLODQuadTree` owns the tree for one terrain; to support multiple terrain
  tiles, instantiate one tree per tile.
- `Select()` clears and refills `out` every frame; callers should pass a
  pre-allocated `std::vector` to avoid repeated allocations.
- LOD distances are a function of `max_triangles`, `lod_count_`, and
  `meters_per_texel_`.  For very flat terrains `mpt` is small and the inner
  LOD rings are very tight; for large-scale landscapes the budget knob has
  a meaningful effect.
- The next step (issue #285 or similar) should implement a `TerrainRenderer`
  that consumes `std::vector<TerrainPatch>` from `Select()` and submits draw
  calls, interpolating vertex positions in the vertex shader using the `morph`
  factor.
