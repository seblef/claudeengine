# GameTerrain — scene object wrapping TerrainData and TerrainMaterial

**Date:** 2026-05-26  
**Issue:** #292  
**Branch:** feat/game-terrain-gameobject-292

## Summary

Adds `game::GameTerrain`, a `GameObject` subclass that owns a `TerrainData` and
`TerrainMaterial` and registers them with `TerrainRenderer` on scene entry. This
completes the integration of the terrain system into the standard game object
lifecycle.

## Design decisions

### VideoDevice* stored in constructor
`GameObject::OnAddedToScene()` takes no parameters (called from `GameSystem::AddObject()`
without a VideoDevice). `TerrainRenderer::Init()` requires a `VideoDevice*`, so the
pointer is accepted in the constructor and stored as a non-owning member. This is
consistent with how other systems obtain their dependencies at construction time.

### Infinite BBox3
Terrain does not participate in the octree visibility system (as stated in the issue).
`OnWorldTransformUpdated()` is a no-op and the constructor passes a default `BBox3{}`
(empty/zero box) since the terrain transform is never queried for culling.

### MapSerializer no-op
Terrain serialization is not part of this feature scope. `SerializeVisitor::Visit(GameTerrain&)`
is intentionally empty — map files do not yet persist terrain objects.

### game → terrain dependency
`GameTerrain.cpp` uses `TerrainRenderer`, which lives in the `terrain` module. Added
`terrain` to `game`'s `target_link_libraries`. This is safe: `terrain` only depends on
`core` and `abstract`, which are already upstream of `game`.

## Output / notes for next features

- Terrain serialization in `MapSerializer` / `MapLoader` should be addressed in a follow-up issue.
- `TerrainRenderer` does not have a `Shutdown()` method yet — `OnRemovedFromScene()` only
  detaches the material (`SetMaterial(nullptr)`). A proper `Shutdown()` on `TerrainRenderer`
  would be needed before `Init()` could be called again (e.g. loading a second map with terrain).
- The `game` CLAUDE.md should be updated to document `GameTerrain`.
