# SelectionTool — Issue #508

## Summary

Extracted object-selection logic from `EditorViewport` into a dedicated
`SelectionTool` class that implements `EditorToolBase`. The viewport now
delegates picking, bounding-box drawing, and Delete-key deletion to the
tool rather than handling them inline.

## Key decisions

### `gizmo_was_using` in EditorToolContext

`SelectionTool::OnRender()` must skip picking on the frame a gizmo drag
ends (LMB is released but the gizmo was still active last frame). The
existing guard `gizmo_was_using_` lives in `EditorViewport` and is updated
**after** the pick check — meaning the correct value is the one from the
**previous** frame.

Adding `gizmo_was_using` to `EditorToolContext` was the cleanest approach:
the viewport builds a fresh context for every `OnRender()` call, passing
its saved `gizmo_was_using_` so SelectionTool can check it without
coupling to viewport internals.

### Context refresh on `SetCommandHistory`

`SelectionTool::OnEvent()` (Delete handler) needs `history_` cached from
`OnActivate()`. Because `SetCommandHistory()` is called after `SetScene()`
in `EditorWindow`, the initial activation (triggered by `SetScene()`) sees
a null history. `SetCommandHistory()` now deactivates then reactivates the
current base tool so its cached context is always up to date.

### `selection_active_` guard kept in `EditorViewport`

The existing `selection_active_` flag (false in placement/creation mode)
is still used as the outer gate before calling `active_tool_base_->OnRender()`.
This preserves the original behaviour: no picking fires while a mesh/light
is being placed. The SelectionTool itself does not need to know about this
flag.

### Helper functions moved to SelectionTool.cpp

`DrawBBoxWireframe`, `ScreenDistToSegment`, `ProjectToScreen`, and
`LightPickDistPx` were all in `EditorViewport.cpp`'s anonymous namespace
and exclusively served the extracted methods. They are now in
`SelectionTool.cpp`'s anonymous namespace. Their implementations are
unchanged.
