# PlacementTool (issue #510)

## What changed

Extracted all object-creation preview and placement logic from `EditorViewport`
into a reusable `PlacementTool` class under `src/editor/tools/`.

### New files

| File | Role |
|---|---|
| `src/editor/tools/ViewportRaycast.h/.cpp` | Free function `ComputeTerrainHit()` extracted from `EditorViewport`; shared with any future tool that needs floor/terrain raycasting |
| `src/editor/tools/PlacementTool.h/.cpp` | Generic placement tool: preview follows cursor, LMB click confirms and pushes `PlaceObjectCommand`, `on_done` fires to switch back to `SelectionTool` |

### Modified files

| File | Change |
|---|---|
| `EditorToolContext.h` | Added `const terrain::TerrainData* terrain_data` field so placement (and future) tools can raycast the terrain without a direct viewport back-reference |
| `EditorViewport.h/.cpp` | Removed `SetPendingMeshTemplate`, `SetPendingLightType`, `SetPendingPlayerStart`, `SetPendingParticleSystem`, `SetOnPlacementDone`, and all preview private members/methods. `PlaceMeshAt` (drag-and-drop) now calls `ComputeTerrainHit()` from `ViewportRaycast`. Tool `OnRender` is now called regardless of `selection_active_` (condition was `selection_active_ && !sculpt_active_`; changed to `!sculpt_active_`) so `PlacementTool::OnRender` is always reached when it is the active tool. Sculpt raycast uses `ComputeTerrainHit()` from `ViewportRaycast` |
| `EditorWindow.h/.cpp` | Replaced `placement_active_` bool with `std::unique_ptr<PlacementTool> placement_tool_`. On each creation-tool transition, constructs a fresh `PlacementTool` with the appropriate preview object and calls `viewport_->SetActiveTool(placement_tool_.get())`. The `on_done` lambda simply calls `toolbar_->SetActiveTool(kSelection)`; the next frame's tool-change logic calls `OnDeactivate()` and `placement_tool_.reset()` safely |

## Decisions and rationale

### `ViewportRaycast` as a free function, not a method
The raycast is now needed by both `EditorViewport` (drag-and-drop, sculpt) and
`PlacementTool` (preview movement). Extracting to a free function with explicit
parameters (`camera`, `terrain_data`, `mouse_pos`, `image_pos`, `image_size`)
avoids coupling `PlacementTool` to `EditorViewport` internals and makes it easy
to unit-test or reuse in `TerrainSculptTool`.

### `terrain_data` in `EditorToolContext`
`PlacementTool::OnRender` is called with a `const EditorToolContext&`. Adding
`terrain_data` to the context is the minimal-coupling approach — the viewport
already owns `terrain_data_` and can populate the context each frame. No extra
parameter or back-pointer needed.

### `OnDeactivate` cleans up the preview
If the user switches tools while a preview is visible, `viewport_->SetActiveTool`
calls `OnDeactivate()` which calls `scene_->RemoveDynamicObject(preview_object_)`.
This replaces the old `CancelPreview()` path and centralises cleanup.

### Deferred `placement_tool_.reset()` after `on_done`
`on_done` fires inside `PlacementTool::OnRender()`. Resetting the
`unique_ptr` immediately would destroy the object mid-call. Instead, `on_done`
only sets the toolbar back to `kSelection`; on the next frame, the
`active_tool != prev_tool_` branch calls `viewport_->SetActiveTool(nullptr)` (which
calls `OnDeactivate()`) before `placement_tool_.reset()`.

### `SetActiveTool(nullptr)` before `placement_tool_.reset()`
Ensures `EditorViewport::active_tool_base_` no longer points to the
`PlacementTool` before its memory is freed. The call order is explicit in the
`active_tool != prev_tool_` block.

## Outputs to keep in mind

- `selection_active_` is now effectively unused for gating `OnRender`; it is
  still set from `EditorWindow` and stored, but the condition was relaxed to
  `!sculpt_active_`. A future cleanup could remove `SetSelectionActive` /
  `selection_active_` entirely.
- `PlaceMeshAt` (drag-and-drop flow) stays in `EditorViewport` as specified.
- The issue mentions a future `TerrainSculptTool` sharing `ViewportRaycast`; the
  free-function API is already in place for that.
