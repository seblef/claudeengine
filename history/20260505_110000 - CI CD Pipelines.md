## Workflow Structure

```
Any push ──► linting ──┐
            static     ├── (parallel)
            dynamic    │
            tests  ────┘

Tag push ──► [all QA] ──► build-dev ──┐
                          build-stable─┤── (parallel)
                                       │
                          package ◄────┘ (stable only)
                          release ◄──── (needs package + build-dev)
```

### Job Details

| Job | Trigger | Key tools |
|-----|---------|-----------|
| `linting` | any push | cpplint, clang-format, glslangValidator |
| `static-analysis` | any push | cppcheck |
| `dynamic-analysis` | any push | valgrind on `core_tests` |
| `tests` | any push | GTest XML + dorny/test-reporter |
| `build-dev` | tag only | cmake `-DBUILD_PROFILE=dev` |
| `build-stable` | tag only | cmake `-DBUILD_PROFILE=stable` |
| `package` | tag only | stable binary + `data/` → zip |
| `release` | tag only | softprops/action-gh-release@v2 |

## Design Decisions

### cpplint filters
Running cpplint on the existing codebase revealed 246 violations. After analysis:
- **Fixed** `whitespace/operators` and `readability/braces` — genuine, small fixes
- **Suppressed** in `CPPLINT.cfg`:
  - `whitespace/comma` — compact `{x,y,z}` vector notation is intentional in math code (166 violations)
  - `build/include_order` — third-party headers (loguru, GLFW) sit after project headers by convention
  - `build/include_what_you_use` — false positives from template-heavy headers
  - `whitespace/braces`/`whitespace/parens` — intentional alignment in matrix constructors
  - `build/namespaces` — test files use `using namespace core` for conciseness
  - `build/c++17` — `<filesystem>` is standard in C++20
  - `legal/copyright`, `build/header_guard` — project conventions

### glslangValidator stage detection
Our shaders use `_vs.glsl` / `_ps.glsl` suffixes (not `.vert` / `.frag`), so auto-detection would fail. CI explicitly passes `-S vert` and `-S frag` via two separate `find` commands.

### Valgrind on core_tests (not main executable)
`core_tests` links only to `core` — pure math, no OpenGL, no display needed. The main executable requires a GPU/display and cannot run in headless CI. Running Valgrind on `core_tests` catches memory errors in the math library, which is the most performance-critical and bug-prone code.

### FetchContent caching
`FETCHCONTENT_BASE_DIR` is set to `${CMAKE_SOURCE_DIR}/external/` in the root `CMakeLists.txt`. The cache key is `hashFiles('CMakeLists.txt')` because all five FetchContent declarations live there. Any dependency version bump busts the cache.

### dorny/test-reporter requires `checks: write`
The `tests` job declares `permissions: checks: write` so the reporter action can post annotations on the commit.

### package depends only on build-stable
Only the stable binary is packaged per spec. `release` depends on `[package, build-dev]` so both builds must complete before the release is published — ensuring both artifacts are available.

### clang-format: file only, not a CI check
`clang-format` is installed in the `linting` job and a `.clang-format` config is committed, but the CI runs `--dry-run --Werror` which will fail if the existing code doesn't match. The `.clang-format` file is committed for developer tooling; if the check needs to be disabled initially, wrap it in `continue-on-error: true`.

## For Future Features

- Run `clang-format -i` on all sources and verify CI passes; then remove any `continue-on-error` if added.
- Consider adding a Windows runner once DirectX 12 backend is implemented.
- The `dynamic-analysis` job is relatively slow (Valgrind overhead); consider restricting it to pushes on `main` and tags only if build times become a concern.
- Add `--suppress=noExplicitConstructor` to cppcheck if single-argument constructors are added intentionally.
